# Unipus AI Automator 维护路线图

这份文档按优先级排列后续工程化方向。目标不是让网站自动化“永不失效”，而是让失效可检测、可定位、可复现、可小范围修复。

## P0：先让失败可诊断

当前最大痛点是网页变化、环境异常、题型误判发生时，维护者只能依赖日志和用户描述。P0 的目标是为每次失败留下足够证据。

交付项：

- 自动保存失败页面 HTML。
- 自动保存失败截图。
- 自动保存 URL、页面标题、面包屑、运行模式、策略名、异常堆栈。
- 自动保存到 `.diagnostics/<timestamp>_<reason>/`。
- 敏感信息不写入诊断包，例如账号、密码、Cookie、本地存储。

验收标准：

- 任意策略执行异常时能生成诊断目录。
- 自动模式中单个任务失败时能留下诊断信息，并尽量继续处理后续任务。
- 致命异常时也能留下最后页面状态。

## P1：集中管理选择器

当前 CSS 选择器分散在 `DriverService` 和各策略文件中，网站改版时搜索和修复成本高。P1 的目标是让页面结构变化集中收敛到少数文件。

交付项：

- 新增 `src/selectors.py` 或 `src/page_objects/`。
- 将高频选择器集中命名。
- 为关键选择器提供备用定位方案。
- 标注选择器稳定性：语义定位、业务结构、CSS class、DOM 层级。

验收标准：

- 提交/下一题按钮、题目容器、选项容器、解析页答案等关键选择器不再散落。
- 页面改版时优先修改选择器层，而不是逐个策略文件搜索。

## P2：拆分策略职责

当前策略通常同时负责页面识别、页面提取、AI 调用、填写、提交、缓存。P2 的目标是将题型策略拆成可组合的流程。

目标结构：

- `Detector`：识别题型。
- `Extractor`：页面转结构化题目。
- `Solver`：缓存、AI、规则求解。
- `Filler`：答案回填。
- `Verifier`：提交前校验。

验收标准：

- 新题型可以复用 Solver、Verifier、缓存逻辑。
- 页面结构变化主要影响 Extractor/Filler。

## P3：建立题目数据模型

当前题目数据多为字符串、列表和临时 dict。P3 的目标是让所有策略围绕统一数据模型工作。

建议模型：

- `TaskContext`
- `Material`
- `Question`
- `Option`
- `Answer`
- `SolveResult`

验收标准：

- AI prompt 从结构化模型生成。
- 缓存写入基于模型字段。
- 答案数量、选项范围等校验不再重复写在各策略中。

## P4：升级缓存体系

当前缓存按面包屑路径保存，简单可读，但对课程改名、题目移动、题干小改不够稳。

交付项：

- 题干 hash。
- 缓存 schema version。
- 答案来源：解析页、AI、人工。
- 缓存置信度。
- 缓存迁移脚本。

验收标准：

- 缓存能识别“路径相同但题目变化”的情况。
- 缓存文件升级不会破坏旧数据。

## P5：命令行和运行模式产品化

当前主要靠交互式 input。P5 的目标是让运行可复现、可脚本化。

建议参数：

- `--auto`
- `--manual`
- `--fast-cache`
- `--headless`
- `--course`
- `--dry-run`
- `--diagnostics-dir`

验收标准：

- 常用模式无需人工输入菜单。
- 失败任务可以用同一组参数复现。

## P6：测试与回归样本

网站自动化很难靠纯单元测试覆盖，但可以依赖真实页面 fixture。

交付项：

- 保存典型题型 HTML fixture。
- Extractor 单测。
- Prompt 生成单测。
- AI JSON 解析单测。
- 曾经失败过的页面进入回归样本。

验收标准：

- 修一个选择器时能验证至少一批历史页面。
- 新题型策略有最小 fixture 覆盖。

## P7：发布工程

部署问题会直接影响用户体验。P7 的目标是让 Lite、Portable、更新包都有可验证的构建过程。

交付项：

- 打包前工作区检查。
- 打包后 smoke test。
- 发布包内写入版本和构建信息。
- 运行时健康检查：Python、依赖、Playwright、FFmpeg、Piper、模型。

验收标准：

- 发布包能在干净环境中完成最小启动验证。
- 依赖损坏时能自动修复或给出明确处理建议。
